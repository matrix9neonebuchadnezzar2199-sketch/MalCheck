RE 解析パイプライン（表層 + 静的）

動的解析は REMnux / 外部ラボ任せとし、MalCheck は PE のパック検出 → 自動アンパック → Ghidra と 耐解析・難読化シグナルのレポート統合 に集中します。

フロー

flowchart TD
    sample[PE_sample] --> surface[Phase1_surface]
    surface --> gate{packer_suspected}
    gate -->|yes| unpack[Phase1b_unpack]
    gate -->|no| ghidraIn[Ghidra_input]
    unpack -->|completed| ghidraIn[unpacked_PE]
    unpack -->|failed| ghidraIn[original_PE]
    ghidraIn --> static[Phase3_Ghidra]
    surface --> report[Report_2_1]
    unpack --> report
    static --> report

Phase 1b — アンパック

実装: mau/unpack_stage.py · 設定: phases.unpack in compose/config/analyzer.yaml

項目内容
トリガーDIE / 高エントロピー / pefile 高エントロピーセクション / packer.detected
ツール順UPX (upx -d) → unipacker（オーケストレータに pip 導入時）
成功時phase1b_unpack.unpacked_path を Ghidra 入力に、oep_rva を MAU_OEP_RVA で headless に渡す
失敗時元 PE で静的解析継続（パイプラインは落とさない）
成果物展開 PE は results/unpack/ のみ（リポジトリにコミットしない）

表層 — 耐解析・難読化

  • format_scanners.py: TLS、anti-debug imports、ASLR/DEP 無効、パッカーっぽいセクション名
  • analyze.py: capa の anti-analysis / anti-vm 要約（capa_anti_analysis）
  • analyze.py: パック疑い PE で FLOSS（未インストール時は success=false で継続）

静的 — decompile_insights ロールアップ

  • scripts/ghidra/auto_analyze.py: 関数別 decompile_insights（anti-debug、goto 平坦化等）
  • mau/re_insights.py: プログラム全体の anti_analysis_signals / obfuscation_stats
  • レポート: re_analysis セクション + verdict 加点

限界（期待値の調整）

  • 全パッカーの自動 OEP は保証しない（UPX / unipacker 対応範囲のみ）
  • 偽 OEP・多重パックは手動 RE が必要なことがある
  • FLOSS / unipacker は surface / orchestrator イメージの再ビルドが必要
  • .NET（de4dot）・ELF 専用パスは本マイルストーンの対象外
